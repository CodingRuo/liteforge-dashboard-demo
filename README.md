# LiteForge Dashboard Demo

A reactive server monitoring dashboard built with [LiteForge](https://liteforge.app) — a signals-based frontend framework with no Virtual DOM.

[![Open in StackBlitz](https://developer.stackblitz.com/img/open_in_stackblitz.svg)](https://stackblitz.com/github/SchildW3rk/liteforge-dashboard-demo)

---

## What is this?

A **demo/showcase** app that simulates 5 servers producing real-time metrics. It demonstrates LiteForge's signal-based reactivity in a high-update-frequency scenario — CPU/RAM charts, live log streams, and KPI cards all update reactively without full re-renders.

**No real backend.** All data is generated by a `SimulationService` that models realistic server load, incidents (CPU spikes, memory leaks, request storms), and recovery events.

---

## Tech Stack

- [`@liteforge/core`](https://liteforge.app/core) — signals, computed, effects
- [`@liteforge/runtime`](https://liteforge.app/runtime) — JSX, components
- [`@liteforge/router`](https://liteforge.app/router) — client-side routing
- [`@liteforge/store`](https://liteforge.app/store) — global metrics store
- [`@liteforge/toast`](https://liteforge.app/toast) — alert notifications
- **Tailwind CSS** — utility classes only
- **Raw SVG charts** — no charting library

## Routes

| Route | Description |
|---|---|
| `/overview` | KPI cards, CPU/RAM charts, server table |
| `/servers` | Server cards with mini sparklines |
| `/servers/:id` | Detail view with 60-tick charts, processes, alerts |
| `/logs` | Live log stream (INFO / WARN / ERROR) |

---

## No charting libraries

This demo intentionally avoids libraries like Chart.js or D3 — all charts are raw SVG updated directly by LiteForge signals, demonstrating fine-grained DOM updates without Virtual DOM overhead. A `<polyline points={...}>` attribute is a signal — when the history array grows by one element, only that single attribute is updated in the DOM.

---

## Simulation

The `SimulationService` (see `src/services/simulation.ts`) runs on a configurable tick interval (250ms / 500ms / 1s). Each tick:

1. Updates all 5 servers with randomized metrics around a baseline load
2. May trigger an **incident** on a random server: CPU spike, memory leak, request storm, or degraded state
3. Fires toast notifications when servers cross critical/degraded thresholds
4. Emits realistic log entries (INFO / WARN / ERROR) per server
